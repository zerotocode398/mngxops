# MngxOps 项目需求文档 - 总览

> **文档版本**: v1.0  
> **创建日期**: 2026-07-10  
> **项目名称**: MngxOps - 多节点 Nginx 配置管理平台  
> **技术栈**: Django 4.2 + Bootstrap 5.3 + SQLite + Paramiko (SSH)

---

## 1. 项目概述

### 1.1 项目背景

目前市场上 nginx 管理工具（如 nginx-ui、nginx-proxy-manager 等）通常只能管理单台机器上的 nginx 实例。当需要管理大量服务器上的 nginx 时，需要在每台机器上部署对应的管理工具，运维成本显著增加。

MngxOps 旨在通过 **集中式 Web 管理平台 + SSH 远程管理** 的方式，实现从单一控制台管理多台服务器上的 nginx 配置、发布、升级等功能，大幅降低多节点 nginx 运维的复杂度。

### 1.2 项目定位

面向 **Linux 运维工程师** 的轻量级、实用型 nginx 集中管理工具，强调操作的可靠性和可追溯性，兼顾易用性。

### 1.3 核心目标

| 编号 | 目标 | 说明 |
|------|------|------|
| G1 | 多节点集中管理 | 一个平台管理所有 nginx 服务器节点 |
| G2 | 配置版本管理 | nginx 配置文件的创建、编辑、版本控制、差异对比 |
| G3 | 配置同步发布 | 将配置安全地推送到远程节点并执行 reload |
| G4 | Nginx 编译升级 | 远程编译安装/升级 nginx，支持自定义模块和编译参数 |
| G5 | 操作可追溯 | 完整的审计日志、登录日志、发布历史 |
| G6 | 权限精细化 | 基于角色的权限控制（RBAC），支持用户组 |
| G7 | 参数可配置 | 平台运行参数可通过页面配置，无需修改代码 |
| G8 | 批量操作 | 支持批量测试连接、批量发布等异步任务 |

## 4. 技术架构概览

### 4.1 后端技术

| 组件 | 技术 | 用途 |
|------|------|------|
| Web 框架 | Django 4.2 | 整体后端 |
| 数据库 | **SQLite** | 唯一数据存储（轻量部署，免去独立数据库服务依赖） |
| SSH 通信 | Paramiko 5.0 | 远程节点管理 |
| 加密 | Cryptography (Fernet) | SSH 凭证加密存储 |
| 异步任务 | Threading + ThreadPoolExecutor | 批量操作异步执行 |
| 认证 | Django Auth + Session | 用户认证与会话 |

### 4.2 关于 SSH 通信库的选择

当前项目使用 **Paramiko** 作为 SSH 通信库。Paramiko 是 Python 生态中最成熟的 SSH 库，优势在于社区活跃、文档丰富、API 易用。但其纯 Python 实现的特性在以下方面存在瓶颈：

| 维度 | Paramiko 现状 | 潜在问题 |
|------|--------------|---------|
| 并发连接 | 依赖 `ThreadPoolExecutor`，每连接一个线程 | 节点 > 50 时线程开销显著 |
| 大文件传输 | 基于 SFTP 逐块读写 | 传输 nginx 源码包（~1MB+）时吞吐不如原生 |
| 批量命令 | 逐连接执行，无法复用会话 | 批量 `nginx -V` 采集时延迟累加 |

**推荐演进路径**：

| 阶段 | 方案 | 适用场景 |
|------|------|---------|
| **当前阶段** | Paramiko + ThreadPoolExecutor | 节点数 ≤ 50，功能开发阶段，保持简单 |
| **优化阶段** | Paramiko + `asyncio` + `asyncio.to_thread()` | 提升批量操作并发效率，无需更换库 |
| **进阶阶段** | [asyncssh](https://github.com/ronf/asyncssh) | 节点数 > 50，原生 async/await，内置 SFTP 并发传输 |
| **替代方案** | [parallel-ssh](https://github.com/ParallelSSH/parallel-ssh) | 纯批量场景，基于 `ssh2-python`（C 扩展），性能最优 |

**建议**：现阶段保持 Paramiko，在系统设置中通过 `node.batch_max_count` 控制并发数（当前硬编码 `3`），待节点规模增长后再评估切换到 `asyncssh` 的成本（需改造 `utils/ssh.py` 中的所有阻塞调用）。

### 4.3 关于数据库的选择

项目**仅使用 SQLite**，不考虑 MySQL。理由如下：

- **部署零依赖**：SQLite 内嵌于 Python，无需安装和配置独立的数据库服务
- **运维简单**：单文件存储，备份/迁移只需复制 `db.sqlite3` 文件
- **规模匹配**：MngxOps 面向中小规模运维团队，节点数通常在百台级别，SQLite 的并发写入能力完全可满足
- **Django 兼容**：Django ORM 屏蔽了底层差异，若未来确实需要迁移到 MySQL/PostgreSQL，仅需修改 `settings.py` 的 `DATABASES` 配置

### 4.4 前端技术

| 组件 | 技术 | 用途 |
|------|------|------|
| UI 框架 | Bootstrap 5.3 | 页面布局与组件 |
| 图标 | Bootstrap Icons 1.11 | 图标库 |
| 交互 | Vanilla JavaScript + Ajax | 异步交互 |
| 模板 | Django Templates | 服务端渲染 |

### 4.5 部署模式

- 单机部署 Django 应用
- 通过 SSH 连接所有目标 nginx 节点
- 无需在目标节点安装 Agent

- 当前活跃的二级菜单高亮，其父级一级菜单自动展开

## 7. 全局设计原则

### 7.1 一致性原则

- 所有列表页面使用统一的表格样式、分页组件、搜索筛选
- 所有表单页面使用统一的表单布局、验证提示、按钮样式
- 所有弹窗使用统一的自定义弹窗组件（非浏览器默认 confirm/alert）
- 所有操作反馈使用 toast 消息提示（统一位置、样式、动画）

### 7.2 可靠性原则

- 所有远程操作前必须进行 SSH 连接测试
- 配置发布前必须进行 `nginx -t` 语法检查
- 配置发布必须包含远程备份步骤
- 所有远程操作记录完整日志

### 7.3 安全性原则

- SSH 凭证使用 Fernet 对称加密存储
- 密码/私钥明文不出现在日志中
- 关键操作需要二次确认
- 敏感操作记录审计日志

### 7.4 异步任务原则 ⚠️ 核心约束

**所有涉及 SSH 远程连接的操作，必须走异步任务机制**，严禁在 HTTP 请求-响应周期内同步执行阻塞式 SSH 调用。

| 操作类型 | 触发方式 | 任务中心 | 进度追踪 |
|---------|---------|---------|---------|
| 节点连接测试（单个） | Ajax POST（同步，10s 超时） | ❌ | 弹窗 loading |
| 节点连接测试（批量） | Ajax POST → 创建 `TaskCenterTask` | ✅ | ✅ |
| 节点系统信息采集 | Ajax POST → 创建 `TaskCenterTask` | ✅ | 单步完成 |
| Nginx 版本检测 | Ajax POST → 创建 `TaskCenterTask` | ✅ | 单步完成 |
| `nginx -V` 编译参数采集 | Ajax POST → 创建 `TaskCenterTask` | ✅ | 单步完成 |
| 凭证启用批量测试 | Ajax POST → 创建 `TaskCenterTask` | ✅ | ✅ |
| 配置发现（远程扫描） | Ajax POST → 创建 `TaskCenterTask` | ✅ | ✅ |
| 配置同步到远程节点 | Ajax POST → 创建 `TaskCenterTask` | ✅ | ✅ |
| 配置发布（备份→上传→校验→检查→reload） | Ajax POST → 创建 `TaskCenterTask` | ✅ | ✅ |
| Nginx 编译升级 | Ajax POST → 创建 `TaskCenterTask` | ✅ | ✅ |
| Nginx 升级回滚 | Ajax POST → 创建 `TaskCenterTask` | ✅ | 单步完成 |

**设计理由**:
- SSH 远程操作耗时不可控（网络延迟、命令执行、文件传输），同步执行会阻塞 Django worker 线程，导致其他用户请求无响应
- 统一走 `TaskCenterTask` 便于追踪执行结果、查看完整日志、失败重试
- 所有异步任务通过 `ThreadPoolExecutor` 执行，最大并发数由系统设置 `release.max_parallel_tasks` 控制（默认 `3`）

**API 交互模式**:
```
# 前端发起异步操作
POST /nodes/batch-test/  →  201 OK
{
  "success": true,
  "async": true,
  "task_center_id": 42,
  "task_center_detail_url": "/releases/task-center/42/"
}

# 前端轮询进度（间隔由 system.task_progress_poll_interval 配置，默认 2 秒）
GET /releases/api/task-progress/42/  →  200 OK
{
  "status": "running",
  "progress": 60,
  "detail": "执行中：成功 2，失败 0，已完成 3/5",
  "result": null
}
```

**前端交互规范**:
- 发起异步操作后立即弹窗展示任务进度（非阻塞，允许关闭后台执行）
- 提供"跳转到任务中心查看详情"链接
- 任务完成后 Toast 通知结果（成功/失败/部分成功）

### 7.5 用户体验原则

- 减少页面跳转，尽量使用弹窗/抽屉完成操作
- 异步操作创建后立即返回任务 ID，前端通过轮询展示进度条（轮询间隔可配）
- 列表页面支持记住搜索条件和分页状态
- 错误信息清晰明确，提供可操作的解决建议
